# 合同台账软件 · 架构设计与选型决策文档

> 版本：v1.0（决策评审稿）｜日期：2026-07-14｜面向读者：行业市场组销售/决策者
> 本文档综合 7 个方向的调研，做过取舍去重，给出**单一明确推荐**，可直接作为开发蓝图。

---

## 1. 台账在做什么（业务本质一段话）

这套台账的本质，是**把 8 名销售各自手里的"合同/商机漏斗"，按统一口径记录下来，并沿三条轴线做汇总与对标**：① 沿"合同层级漏斗"（已签约→待签约→洽谈中→项目线索→培育中）看管道存量；② 沿"1–12 月时间轴"看到款节奏；③ 沿"收入/利润/成本三类目标"看每个人和团队"距目标还差多少、完成率多少、比上期变好还是变差"。约 441 条记录、8 名销售，数据天然**按销售人员分区**（每人只改自己那一摊），最终要**在无互联网、可能无局域网的纯本地环境下汇总到经理**。因此它不是一个 CRM，而是一个**离线、加密、可审计的"财务口径台账 + 目标达成看板 + 文件摆渡式协同"系统**。

## 2. 需求与硬约束提炼

### 2.1 硬约束（不可妥协，决定选型权重）

| 约束 | 含义 | 对设计的影响 |
|---|---|---|
| **100% 纯本地、无外网出口** | 安装即用，运行期零网络请求，无 CDN/无在线运行时下载 | 排除一切依赖服务端的同步引擎；所有依赖必须离线内嵌打包 |
| **可能无局域网/无共享盘** | 连中转文件夹都不一定有 | 协同只能靠**文件物理摆渡（U盘/加密邮件附件）** |
| **Windows 平台** | 客户 100% Windows 11 | 跨平台能力是浪费；优先 Windows 原生成熟度 |
| **国企数据安全** | 数据落地加密、防篡改、可审计、可追责 | 全库加密 + 登录鉴权 + 数据包签名 + 审计日志，均为必备而非可选 |
| **"系统级"能力** | 不是脚本，要能长期运行维护 | 需正规架构分层、可交接（人才储备）、可签名分发 |

### 2.2 核心功能诉求

- **录入**：复刻/内嵌 Excel 手感，支持从现有 xlsx 批量导入、整块粘贴、公式自动计算、下拉校验。
- **统计**：按 `销售人员 × 合同层级` 分组求和，算合计、与指标差距(gap)、完成率、环比上期。
- **可视化**：经理总览页 + 销售个人页；漏斗、完成率、月度到款曲线、排行榜。
- **协同**：8 个销售 → 经理汇总 → 可回下发；去重、冲突裁决、审计留痕。

### 2.3 关键难点（本项目胜负手）

> **在无网无盘的前提下实现"各管一摊 → 汇总到经理"的协同。** 这是全项目最难、最需要提前拍板的部分，第 6 节专门详述。

## 3. 推荐技术栈（对比结论）

### 3.1 桌面框架对比结论

| 维度（按本项目权重） | **.NET WPF (推荐)** | Electron+Univer | Tauri | Python PySide6 | Flutter |
|---|---|---|---|---|---|
| 离线U盘分发/体积 | ★★★★★ 单exe自包含 | ★★ 80–150MB | ★★★★ 但依赖WebView2 | ★★★ 包大易误报 | ★★★★ |
| 国企合规/代码签名 | ★★★★★ MSIX+Authenticode | ★★★ | ★★★★ | ★★ 易反编译 | ★★★ |
| 内嵌类Excel手感 | ★★★★ | ★★★★★ Univer最强 | ★★★★ | ★★★ | ★★★ |
| Excel读写互操作 | ★★★★★ NPOI/ClosedXML | ★★★ | ★★★ | ★★★★★ pandas | ★★ |
| 加密SQLite落地 | ★★★★★ | ★★★★ | ★★★★★ | ★★★★ | ★★★ |
| Windows人才/可维护 | ★★★★★ | ★★★ | ★★★ 需Rust | ★★★ | ★★★ |
| **加权总评** | **4.7** | 3.4 | 3.7 | 3.4 | 3.1 |

### 3.2 单一推荐技术栈

> **主栈：.NET 8 + WPF (C#) + WebView2 内嵌 Univer + SQLite(SQLCipher) + NPOI**

一句话理由：本项目真正的胜负手是**离线单exe分发、代码签名合规、Excel互操作、加密落地、长期可维护**这五项，.NET 在这五项上全部拿满；再通过 **WebView2 内嵌 Univer** 把唯一的短板（类 Excel 录入手感）补齐——既拿到原生桌面的分发/签名/安全优势，又拿到 Web 系电子表格最强的录入体验。

| 层 | 选型 | 许可证 | 备选 |
|---|---|---|---|
| 桌面壳 | WPF (.NET 8) | MIT | WinUI 3（想要现代 UI 时） |
| 录入网格 | **Univer**（WebView2 内嵌，本地打包） | Apache-2.0 | 原生 DataGrid / Jspreadsheet CE(MIT) |
| xlsx 读写 | **NPOI** | Apache-2.0 | ClosedXML(MIT)；**避开 EPPlus(商用收费)** |
| 数据库 | **SQLite** | Public Domain | — |
| 加密 | **SQLitePCLRaw.bundle_e_sqlcipher** (AES-256) | 混合(MIT/BSD) | SQLite3MC(MIT) |
| 图表 | **Apache ECharts**（WebView2 内嵌） | Apache-2.0 | 原生 LiveCharts2 |
| 透视表 | AntV S2（可选，做汇总透视） | MIT | 原生表格 |

**为什么不选各方案**：
- **Electron+Univer**（备选一）：录入体验最强，但 80–150MB 体积 + 更大攻击面让国企安全审计更繁琐。仅当"内嵌 Excel 手感"被领导定为一票否决诉求时才整体切它。
- **Tauri**：安全体积俱佳，但依赖系统 WebView2；若机群混有隔离 Win10 机器，离线 bootstrap WebView2 会破坏"U盘即用"。作为第三备选。
- **Python PySide6**（备选二）：**定位为 2–3 周快速原型**验证数据模型与汇总口径最快；但字节码易反编译、PyInstaller 打包杀软误报率高，不建议作为国企正式交付。
- **Flutter**：跨平台优势本项目用不上，Excel 生态与合规成熟度弱，**不推荐**。

> 关于 WebView2 依赖的说明：Windows 11 已内置 WebView2 Runtime，用户机器满足。若未来发现有隔离 Win10 机，需随安装包附带 WebView2 离线 bootstrapper。若这是硬阻断，则回退到"纯 .NET 原生 DataGrid + LiveCharts2"方案（牺牲部分 Excel 手感换取零 WebView2 依赖）。

## 4. 整体架构（分层）

```
┌─────────────────────────────────────────────────────────────────┐
│  表示层 (Presentation)  —— WPF 窗体 + WebView2 宿主                │
│   · 导航壳(WPF)：登录、菜单、视角切换、向导                          │
│   · 录入区：WebView2 内嵌 Univer（类Excel网格，本地打包）            │
│   · 看板区：WebView2 内嵌 ECharts（漏斗/仪表盘/曲线/排行）           │
│   · 经理透视：AntV S2 透视表                                        │
├─────────────────────────────────────────────────────────────────┤
│  业务层 (Domain / Service)  —— C# 领域服务                          │
│   · 校验引擎(U=S−T、V=Σ月度、必填、枚举、层级联动)                   │
│   · 统计引擎(合计/gap/完成率/环比/漏斗/月度)                         │
│   · 导入导出服务(NPOI xlsx ↔ 领域模型 ↔ SQLite)                     │
│   · RBAC 权限过滤(查询强制注入 owner_user_id)                       │
├─────────────────────────────────────────────────────────────────┤
│  同步层 (Sync)  ★本项目核心                                         │
│   · 数据包引擎：导出 .cvpk(加密+签名) / 导入合并向导                 │
│   · 合并引擎：SQLite Session changeset upsert + 冲突裁决队列          │
│   · 审计/历史：事件日志(events.ndjson) + 可选 git bundle 回溯         │
├─────────────────────────────────────────────────────────────────┤
│  数据层 (Data)  —— 每销售一个独立加密 SQLite 库                      │
│   · Contract / MonthlyPayment / Target / Snapshot / SyncPackage    │
│   · AuditLog(append-only) / Enum* 字典表                            │
│   · 快照(Online Backup API) + 软删除回收站                          │
├─────────────────────────────────────────────────────────────────┤
│  安全层 (Security, 横切)                                            │
│   · 全库 AES-256(SQLCipher)  · Argon2id 登录  · DPAPI 密钥保护       │
│   · age/Ed25519 数据包加密+签名  · 审计哈希链  · 整库加密快照         │
└─────────────────────────────────────────────────────────────────┘
                    拓扑：Hub-and-Spoke（经理=hub）
   销售A库 ─┐
   销售B库 ─┼─→ [.cvpk 加密包 · U盘/邮件摆渡] ─→ 经理主库(唯一合并点) ─→ 回下发
   ...    ─┘
```

**关键架构决策**：
1. **主库只存在于经理机**，共享盘（若有）仅做文件中转，绝不多写单库——规避 SQLite 网络锁损坏风险。
2. **每销售一个独立加密库**（物理隔离 + 天然分区），合并本质是"拼图"而非"抢锁"。
3. **表示层与业务层解耦**：录入/图表用 Web 组件（WebView2），业务/数据/安全全在 C# 侧，Web 层不碰密钥、不碰 xlsx 解析。

## 5. 数据模型蓝图

### 5.1 核心表与关系

```
Team 1──* User 1──* Contract *──1 Enum(stage/product/currency…)
                       │
                       ├─1──* ContractMonthlyPayment  (月度到款，1-12月时间轴)
                       └─1──* ContractSnapshot         (每上报期指标快照→环比)
User/Team 1──* Target                 (销售×年 / 团队×年 三类指标)
ReportingPeriod 1──* ContractSnapshot / SyncPackage
AuditLog(append-only, 全表变更留痕)
```

### 5.2 设计要点

- **金额一律用 `INTEGER`（分）**存储，严禁浮点，避免统计误差。
- **`profit_est_u = S − T`** 用 SQLite 生成列（GENERATED STORED）强制；**`payment_est_v = Σ月度`** 用触发器维护。二者杜绝手算错误。
- **`contract_uid` 全局唯一**（`{owner_user_code}-{seq}` 或 UUID），是跨库合并去重的稳定主键；合同编号可空（线索/培育期无编号）。
- **月度到款拆长表**：`period_month=4` 存"1-4月累计"(`is_cumulative=1`)，5–12 各存单月；`kind` 区分"预计/实际"，一表两用。
- **快照表实现环比**：每期冻结当期指标，本期对上期按 `contract_uid` LEFT JOIN 求差，历史口径永久冻结、可复核、可追溯漏斗层级迁移。
- **统一审计字段**：`created_at/by、updated_at/by、row_version、is_deleted`（软删除）。

### 5.3 Excel 列映射摘要

| 中文列 | 字段 | 归属表 | 录入/计算 |
|---|---|---|---|
| 合同层级 | stage_code(FK) | Contract | 录入 |
| 销售人员 | sales_user_id / owner_user_id | Contract | 录入 |
| 销售合同金额 / S / T | contract_amount / revenue_est_s / cost_est_t | Contract | 录入(分) |
| **业务利润 U** | profit_est_u | Contract | **计算 U=S−T** |
| **预计到款 V** | payment_est_v | Contract | **计算 V=Σ月度** |
| 1-4月累计 / 5–12月 | amount(month=4,is_cumulative=1 / 5..12) | ContractMonthlyPayment | 录入 |
| 截止累计到款 | received_to_date | Contract | 录入 |
| 培育来源 | cultivation_source | Contract | 录入(培育中必填) |
| 收入/利润/成本控制指标 | revenue_target/profit_target/cost_ceiling | Target | 录入 |
| 产品类型/币种/项目属性/市场领域/板块 | *_code(FK) | Contract | 录入(枚举) |

> 枚举用 `code(英文短码)` 做外键、`name` 供显示；市场领域/业务板块做成 admin 可维护字典（年度口径会变），层级/产品类型/项目属性相对固定。

## 6. ★离线协同工作流设计（本项目最关键）

### 6.1 选型结论：放弃一切服务端同步引擎

无网无盘硬约束**直接排除** ElectricSQL / PowerSync / Zero / Triplit / Turso Sync（全部需中心 Postgres/服务端）。也**不首选纯 CRDT**（Automerge/Yjs）：本场景并发极低、财务数字必须"看得见的人工裁决"而非静默 LWW，CRDT 的自动收敛优势用不上反增语义复杂度。

> **推荐主线 = 数据包底座(sneakernet) + SQLite Session changeset 合并 + 事件日志审计。**
> 拓扑为 **Hub-and-Spoke**（经理为唯一 hub，8 个销售只与经理交换，不互相交换），冲突面从 O(n²) 降到 O(n)，且数据天然按 `销售人员` 分区，绝大多数合并是"无交集归并"。

### 6.2 数据包 `.cvpk` 格式

加密 zip 容器，结构：

```
manifest.json    # 包类型(full/delta)、导出人、导出时间、schema版本、
                 # base_master_version(基于哪个主版本)、记录数、SHA-256
payload.bin      # SQLite Session changeset（增量）
events.ndjson    # 字段级变更事件(谁/何时/从多少改到多少/基于哪版)——审计层
signature        # Ed25519 签名，防篡改防冒充
外层: AES-256-GCM 加密，经理↔各销售一对一密钥(防止销售A能解销售B的包)
```

### 6.3 合并 / 去重 / 冲突 / 审计

- **去重三道防线**：① `contract_uid` 主键唯一；② UI 限制"只能改自己名下合同"，数据按 owner 分区；③ 合并时同主键多来源 → **强制进入冲突裁决队列，不静默合并**。
- **冲突裁决规则**：
  - 同一合同不同销售改 → 按 `合同编号` **归属人优先** + 记冲突日志人工复核；
  - **金额字段冲突 → 一律弹出人工裁决**（三方对比：主库值/来包值/建议值），绝不静默覆盖；
  - 其余字段 → LWW（按 `row_version`/时间戳取新）。
- **审计**：所有导入的原始 `.cvpk` 永久归档(append-only)；`events.ndjson` 保证任一数字可回答"谁、何时、从多少改到多少、基于哪版"；（可选）经理端主库每期规范化导出为按销售拆分的 NDJSON 并 git 提交，用 `git log/diff` 提供任意两期对比与回滚。

### 6.4 双向同步闭环（含"下发后销售又改了"）

```
1. 下发：经理产出 master@vN(全量加密包) → 分发给 8 人
2. 销售导入 master@vN；此后本地改动全部记为"基于 vN 的 delta"
3. 汇总：销售导出 delta(base=vN) → 经理按 uid 归并，冲突进裁决队列
4. 经理生成 master@vN+1，审计留痕 → 回到步骤1
★ 因 delta 始终标注 base_master_version，经理能识别"该改动基于旧主版本"，
  做三方合并(vN、master@vN+1、销售delta)，绝不用旧值覆盖新主库。
```

### 6.5 有无局域网两种模式

| 环节 | 模式A：无局域网(U盘/邮件) | 模式B：有共享盘 |
|---|---|---|
| 投放 | 导出到U盘/邮件附件，人工交经理 | 导出到共享盘 `\\share\台账\上报\<期次>\` |
| 收集 | 逐个插U盘导入 | 框选目录全部包一次导入 |
| "谁没交" | 系统列应收清单，手工对照 | 系统扫描目录自动比对名册标注缺交 |
| 回下发 | U盘/邮件逐人发 | 写入 `\\share\台账\下发\<期次>\` 供自取 |
| 关键约束 | 强调数据包加密(U盘易丢) | **主库仍只在经理机**，共享盘只做中转不做多写库；销售仅能写自己那份 |

> 若明确永远不引入 git，用自建 append-only 事件日志（Lamport 时间戳确定性重放）即可满足审计。SQLite Session 扩展自 3.13.0 起内置、零外部依赖，是离线 patch/merge 首选。CRDT(Automerge)、Dolt 作为**未来"多人真并发编辑同一表"时的升级路径**，非首选。

## 7. 数据录入 = 内嵌类 Excel 模块方案

### 7.1 组件选型结论

| 需求 | 选型 | 理由 |
|---|---|---|
| 内嵌电子表格网格 | **Univer(Apache-2.0, 13.8k★)** | 唯一同时满足"类Excel全功能+免费商用+活跃维护"；合并单元格、真公式引擎、下拉验证、整块粘贴全支持；纯前端可离线打包 |
| xlsx 读写(后端) | **NPOI(Apache-2.0)** | 国内最广、支持下拉数据验证/公式/样式；不依赖装 Office，纯离线 |
| ~~Luckysheet~~ | 排除 | 已于 2025-08 归档，官方指向 Univer |
| ~~Handsontable~~ | 排除 | 商用必须付费，国企采购成本高 |
| ~~AG Grid~~ | 排除 | Community 无公式/合并/整块粘贴，不是电子表格 |
| ~~EPPlus~~ | 排除 | v5 起商用收费 |

> 架构取舍：**xlsx 解析放 .NET 后端(NPOI)，前端 Univer 只负责网格交互**——避免 SheetJS CE 导出丢下拉验证的坑，也让密钥/校验逻辑集中在 C# 侧受控。

### 7.2 录入功能设计

- **导入现有 xlsx**：拖入 → NPOI 读取 441 行 → 表头自动匹配(记忆映射模板) → 字段映射确认 → 差异预览(新增/更新/异常行) → 事务落库。异常行留"待修正"暂存区不阻断其余导入。
- **批量粘贴**：Univer 原生支持从 Excel 整块复制粘贴；键盘流(Tab/Enter/方向键/填充柄/Ctrl+D)。
- **字段级校验（实时红黄标记）**：
  - 公式勾稽：`U=S−T`、`V=1-4月累计+Σ(5..12月)`，公式列灰显只读实时重算；
  - 枚举合法性：合同层级(5类)/项目属性/产品类型(6类)/币种/销售人员 绑定下拉，非法值高亮；
  - 层级联动必填：已签约→编号+签约日期+金额必填；洽谈/待签约→预计签约日期必填；培育中→仅培育来源必填；
  - 违规单元格红/黄描边 + 行首状态灯(✓/⚠/✗) + 悬停原因(如"V与月度之和不符，差12,000")。
- **导出**：NPOI 反向生成带公式/下拉/按`销售×层级`分组汇总的 xlsx，格式与原台账一致，供上报。

## 8. 可视化仪表盘设计

图表库统一用 **Apache ECharts(Apache-2.0, ~64.8k★)**：漏斗、仪表盘gauge、堆叠柱、环形进度全部原生内置(Chart.js/Recharts 缺漏斗)，dist 自包含单文件天生离线，中文文档最全。透视汇总表用 **AntV S2(MIT)**。

### 8.1 经理总览页（团队视角）

顶部一行 **KPI Scorecard**（左上起，视线首落点）：团队总收入达成率 / 总利润达成率 / 成本控制达成率 / 总到款进度——每格含 `当前值 / 完成率% / 环比上期 / 距目标gap`。

| # | 卡片 | 图表形态 | 说明 |
|---|---|---|---|
| 1 | **销售完成率排行榜**(旗舰,占最大面积) | 横向条形 + 每人内嵌子弹条 | 一屏比完8名销售 |
| 2 | 团队销售漏斗 | ECharts funnel + 层间转化率标签 | 5类层级，金额口径 |
| 3 | 团队1-12月到款 | 月柱 + 累计折线 + 目标节奏基准线 | 当前累计 vs 应达累计 |
| 4 | 销售×合同层级 | 堆叠柱 | 复刻原Excel"按销售×层级求和" |
| 5 | 产品/领域/板块构成 | 堆叠柱或treemap，可下钻 | 6类产品类型下钻明细 |
| 6 | **目标达成矩阵表** | AntV S2 透视表，条件着色 | 行=销售,列=收入/利润/成本,格=合计/gap/完成率/环比 |

### 8.2 销售个人页（朴东旭本人视角）

首屏三个大 **环形完成率仪表盘**（收入/利润/成本控制）：正中大字"完成率%"，下方小字"距目标还差¥XXX"——直接回应领导诉求。

| # | 卡片 | 图表形态 |
|---|---|---|
| 1 | 个人三指标 | **子弹图**(实际vs目标vs分段基准)——看清差距落在差/合格/优哪个区间 |
| 2 | 个人销售漏斗 | funnel(金额/数量可切换) |
| 3 | 个人1-12月到款 | 月柱+累计折线+目标节奏线——判断进度领先/落后 |
| 4 | 我的合同明细 | Univer/S2 可编辑网格，按层级分组，临近"预计签约日期"高亮 |
| 顶 | scorecard | S/T/U/V/截止累计到款，各带环比 |

### 8.3 可视化通用原则

- 单屏聚焦 5–9 指标；最关键 KPI 放左上；每个 scorecard 给"当前值+环比+距目标"三数。
- **配额达成用子弹图/仪表盘，不用折线**（折线掩盖pacing）；管道用漏斗+转化率，不用饼图。
- 颜色：达标绿/临界黄/欠佳红，但**成本控制指标方向反转（越低越好）配色单独定义**；采用 Okabe-Ito 色盲安全palette，关键状态叠加▲▼图标不只靠红绿；对比度≥3:1。
- 国企桌面场景：字号偏大、少动画、支持一键导出图片/PDF（无网下的汇报出口）。

> 参考的开源 CRM（Twenty/Refine/Krayin/EspoCRM）仅借鉴管道看板与 KPI scorecard 布局，**不搬代码架构**——本项目数据模型更偏财务台账。

## 9. 安全与合规方案

分层安全架构（全部 BSD/MIT/ISC/Apache 宽松许可，无 GPL 传染、无联网依赖、可离线打包，唯一收费项 SQLite SEE $2000 **不必采购**）：

| 层 | 方案 | 组件/许可 |
|---|---|---|
| **存储加密** | 每销售一个独立加密 SQLite 库，全库 AES-256 | SQLitePCLRaw.bundle_e_sqlcipher（.NET链路，成熟） |
| **登录鉴权** | Argon2id 本地校验（内存硬抗爆破，不存明文） | NetDevPack.Security.PasswordHasher(MIT) / libsodium |
| **权限(RBAC)** | 销售/经理/管理员三角色；查询强制注入 `WHERE owner_user_id=当前用户` | 数据访问层统一注入，UI无绕过入口 |
| **密钥保护** | DB主密钥由用户口令经Argon2id派生，不落盘；本地密钥文件用 Windows DPAPI(CurrentUser)保护 | System.Security.Cryptography.ProtectedData |
| **数据包安全** | 先 age(X25519,经理公钥)加密 → 再 Ed25519 签名(确认哪位销售、防篡改) | age(BSD) / minisign(ISC)；或应用内 libsodium 一体化 |
| **审计** | Audit.NET 拦截所有写操作 → append-only 审计库；审计记录 **HMAC 哈希链**(每条含前一条哈希)防抵赖；日志表禁UPDATE/DELETE | Audit.NET(MIT) |
| **韧性/防呆** | SQLite Online Backup API 自动加密快照(每次导出前打点)；软删除回收站；`PRAGMA integrity_check`；结合审计旧值/新值JSON做单条回滚 | 内置 |
| **密钥托管** | 管理员离线保管恢复口令，防DPAPI绑机导致换机数据永久丢失 | 流程制度 |

> 进阶合规项：登录失败锁定与自动锁屏；数据包期次防误用校验；**国密算法(SM4/SM3)可选**以贴合国企自主可控；离线许可/机器绑定。

## 13. 开工前需用户拍板的关键决策点

1. **录入手感 vs 分发安全的优先级**：是否接受 WebView2 依赖以换取 Univer 的最强 Excel 体验？若领导把"内嵌 Excel 手感"定为一票否决，则整体切 Electron+Univer（体积大、审计繁）；若"纯U盘即用/零依赖"优先，则回退纯 .NET 原生 DataGrid（手感略逊）。**推荐：WPF+WebView2+Univer。**
2. **完成率口径**：完成率是**仅"已签约"计入**（严口径），还是"已签约 + 加权在途（洽谈×成交概率）"？这决定 EnumStage 是否要加 `win_probability` 列，直接影响所有达成数字。**必须先定义。**
3. **是否有共享盘**：确认协同是"纯U盘/邮件"(模式A) 还是"有共享盘中转"(模式B)，或两者都要。影响缺交提示与投放路径设计。
4. **机群是否含隔离 Win10 机**：若有，需附带 WebView2 离线 bootstrapper，或考虑回退无 WebView2 方案。
5. **是否引入 git 做审计回溯**：接受则用 git bundle 拿到世界级历史/回滚；否则用自建事件日志（也满足合规）。**推荐：自建事件日志 for MVP，git 作为进阶。**
6. **国密要求**：国企合规是否**强制** SM4/SM3？若强制则加密层需从 AES-256 调整为国密算法（影响加密库选型）。
7. **多币种处理**：非 CNY 合同是**分币种分列统计**（默认，无隐性折算误差），还是需**统一折算**（需维护汇率）？
8. **代码签名证书**：MSIX+Authenticode 需一张企业代码签名证书，由公司 IT 提供还是项目采购？影响 M6 分发。
9. **密钥托管责任人**：DPAPI 绑机绑用户，换机不可解——谁作为管理员离线保管恢复口令？需明确制度，否则有数据永久丢失风险。
10. **经理是否兼背合同**：经理默认是否也有销售身份（视角切换"我的台账/团队汇总"）？影响权限模型。
