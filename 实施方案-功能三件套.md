# 合同台账管理软件 — 四块功能统一实施方案

> 综合四个维度调研（native-wpf-grid / login-rbac-personal / offline-sync / persistence）的落地设计。全程紧扣：**.NET 9 + 原生 WPF、100% 本地离线、U盘/加密邮件摆运、国企数据安全与审计、复用已有 Core 领域引擎（双口径完成率/漏斗/月度 + NPOI 导入）**。

---

## 0. 一处必须先讲清的跨维度冲突（最重要）

四份调研里有一处**结论互相打架**，作为综合方必须先裁决，否则后面全乱：

- 「持久化」维度推荐 **SQLite3MultipleCiphers**，核心理由是"它是唯一自带 SQLite **Session 扩展**的加密库，功能三零额外构建成本"。
- 「离线协同」维度经过更深入的源码核实后**明确否掉 Session**：SQLitePCLRaw 的托管层根本没有 `sqlite3session_*` P/Invoke 绑定；且 Session 语义对"金额字段绝不能静默丢改"是高风险（无主键表被静默忽略、OMIT 冲突时删行等）。

**裁决：以「离线协同」维度为准 — 放弃 Session 扩展，协同改为自研 op-log。**

这个裁决的连锁影响，必须在方案里一次性理顺：

| 受影响项 | 原理由（作废） | 修正后的结论 |
|---|---|---|
| 加密库为何选 SQLite3MC | "唯一自带 Session" | **仍选 SQLite3MC**，但理由换成：MIT 许可（无 GPL 传染，国企友好）、单一原生 DLL（U盘即用）、透明加密整库含 WAL/journal、默认 ChaCha20-Poly1305、与 EF Core 无缝、活跃维护。**这些理由与 Session 无关，选型不变。** |
| 变更集怎么来 | Session changeset | **自研 op-log 表**（普通表，纯托管，无 native 依赖、可审计） |
| `SyncPackages` 表 | 存 Session changeset blob | 改存 op-log 导出的 `.cvpk` 元数据 |

结论：**加密库选型不受影响，仍是 SQLite3MC；但协同机制从 Session 换成自研 op-log。** 下面所有设计按此裁决展开。

## 2. 地基：加密本地持久化

### 2.1 依赖组合（务必用 `.Core` 包避免 bundle 冲突）
```
Microsoft.Data.Sqlite.Core
Microsoft.EntityFrameworkCore.Sqlite.Core
SQLite3MultipleCiphers          // 提供加密原生 provider
BCrypt.Net-Next                 // 密码哈希
```

连接串用 Builder（防注入），`Password` 关键字内部转 `PRAGMA key`；改密用 `PRAGMA rekey`：
```csharp
var cs = new SqliteConnectionStringBuilder {
    DataSource = dbPath,
    Mode = SqliteOpenMode.ReadWriteCreate,
    Password = dbKey                       // 切勿字符串拼接
}.ToString();
```

### 2.2 数据库表结构（对齐 Core 领域模型：S/T/U=S−T、5类漏斗、1-12月到款、指标）

所有业务表继承 `AuditableEntity`，天然满足国企审计留痕与"禁止硬删除"。

| 表 | 关键列 | 说明 |
|---|---|---|
| `Contracts` 合同/商机 | `Id`, `ContractNo`(唯一,upsert键), `RowId`(GUID,协同用), `FunnelStage`(5类漏斗枚举), `SalesPersonKey`(销售业务键), `RevenueS`, `CostT`, `ProfitU`(=S−T,冗余+`CHECK`兜底), `Year`, 审计列, `Version` | 与 Core 的 S/T/U、5类漏斗一致 |
| `MonthlyReceipts` 月度到款 | `Id`, `ContractId`(FK), `Month`(1–12), `Amount`, `RowId` | 1..12 拆行，支撑月度口径 |
| `SalesPersons` 销售 | `Id`, `SalesPersonKey`(唯一), `Name`(朴东旭), `Dept` | 8+ 名销售 |
| `Targets` 指标 | `Id`, `SalesPersonKey`, `Year`, `TargetRevenue`, `TargetProfit` | 支撑双口径完成率/目标达成 |
| `Users` 账户 | `UserId`, `UserName`(唯一), `DisplayName`, `SalesPersonKey`(数据分区核心), `Role`(0=Sales/1=Manager/2=Admin), `PasswordHash`(BCrypt复合串), `MustChangePassword`, `IsActive`, `CreatedAt`, `LastLoginAt` | 登录/权限 |
| `OpLog` 变更日志 | `Seq`(单调整数PK), `Table`, `RowId`, `Op`(upsert/delete), `ColsJson`, `UpdatedAt`(UTC), `UpdatedBy`, `Site`(站点码), `RowVersion` | **协同核心，自研 op-log** |
| `SyncPackages` 同步包账 | `Id`, `Direction`(delta/masterReturn), `Site`, `Period`, `BaseVersion`, `PayloadSha256`, `CreatedAtUtc`, `AppliedAtUtc` | 记 `.cvpk` 摆运/合并 |
| `Imports` 到包记录 | `Id`, `Site`, `Period`, `ImportedUtc` | 算"谁没交" |
| `Sites` 站点/应收 | `Site`, `SalesPersonKey`, `PeriodRequired` | 应收清单基准 |
| `AuditLogs` 审计 | `Id`, `EntityName`, `EntityId`, `Action`, `ChangedBy`, `ChangedAtUtc`, `OldJson`/`NewJson` | 登录/切换/导出/改数据全留痕 |

**关键约定：**
- 金额存 `decimal` 或整数分；`ProfitU` 由 Core 引擎算、DB 存冗余列并加 `CHECK(ProfitU = RevenueS - CostT)`。
- 统计（双口径完成率/漏斗/月度）**仍走 Core 内存引擎**，DB 只做加密持久化 + 变更集，职责不越界。
- 每张业务表带协同五元组：`RowId / UpdatedAt / UpdatedBy / RowVersion / Site`。

### 2.3 EF Core 工程做法
- **软删除**：`SaveChangesInterceptor` 把 `Deleted` 状态改写成 `Modified`（UPDATE 而非 DELETE）+ 全局 `HasQueryFilter(e => !e.IsDeleted)`。契合国企禁硬删。
- **审计字段**：基类 `AuditableEntity{CreatedAtUtc,CreatedBy,UpdatedAtUtc,UpdatedBy,Version}`，拦截器统一填充。可读 `PandaTechAM/be-lib-efcore-audit-base` 抄。
- **乐观并发**：SQLite 无原生 rowversion，用 `Version` 列 + AFTER UPDATE 触发器自增（Brice Lambson 权威做法），冲突自动抛 `DbUpdateConcurrencyException`。
- **迁移**：EF Core Migrations 编译进程序，启动 `Database.Migrate()` 离线自升级；触发器用 `migrationBuilder.Sql()` 写进迁移。
- **首次灌库**：NPOI → `ImportResult`(451条) → 映射实体 → 单事务 `AddRange`+`SaveChanges`（或 `EFCore.BulkExtensions.BulkInsert` 提速）；按 `ContractNo` upsert 保证幂等。灌完导出首个基线 op-log 作 `baseVersion`。

## 3. 功能① 原生 WPF「类 Excel」批量录入

### 3.1 选型：增强原生 DataGrid（不上商用控件）

判断依据：台账是**受约束的结构化录入**（固定列+下拉+计算列+校验），不是自由建表/公式保真。商用电子表格（SfSpreadsheet/DevExpress/Telerik/Spread.NET）解决的恰是你不需要的问题，且**对国企一律收费**（Syncfusion 社区版明确排除政府/准政府机构），还要引闭源 DLL + license，与"U盘摆运、软件合规评审"冲突。

原生 DataGrid **已内置、无需写代码**的能力：键盘流（Tab/Enter/方向键）、`DataGridComboBoxColumn` 下拉枚举列、只读计算列绑 ViewModel 属性、行/列虚拟化（451条毫无压力）、`ClipboardCopyMode` 选区 Ctrl+C 输出 Excel 可识别 TSV。

### 3.2 需自写的 4 件事（都有现成参考代码，约 2–3 人日）

| 能力 | 实现要点 | 参考 |
|---|---|---|
| **Excel 整块粘贴** | `DataGridClipboardBehavior` 解析剪贴板 TSV，按选中起点铺开写入 | `quanSadie/DataGridWPF_copy_paste_cell` + CodeProject 两篇源码 |
| **向下填充** | 一期做 **Ctrl+D**（取上一行同列值写入选中行，几十行代码）；**填充柄拖拽**列二期（需自绘右下角小方块 + 命中测试，成本高） | CommunityToolkit issue #4873（填充柄仍未实现，作设计参考） |
| **单元格级校验红标** | 模型实现 `INotifyDataErrorInfo`，非法值 WPF 自动红框 + `Validation.ErrorTemplate` 定制红标 | BCL 内置 |
| **撤销重做** | MVVM 下自建 `UndoManager`：`Stack<Command>` 记录"单元格旧值→新值"，不依赖控件 | 自写 |

- **下拉列**：`DataGridComboBoxColumn`（合同层级5类、产品类型）。
- **计算列**：只读 `DataGridTextColumn` 绑 ViewModel 计算属性（`U=S−T`、预计到款=月度之和）——比在电子表格里写公式更可控、更好审计。
- **地基**：直接引 `dotnet/DataGridExtensions`（MIT，⭐906，dotnet 组织维护，官方文档即含 Excel-like copy/paste，透明附着无需继承）。

## 4. 功能② 登录 / RBAC / 销售个人达成页

### 4.1 密码哈希与登录
- **BCrypt.Net-Next**（MIT，纯托管单 DLL）：`BCrypt.HashPassword` / `BCrypt.Verify`，工作因子调 12。避开任何依赖 libsodium 原生 dll 的方案（离线 U 盘会缺 dll）。仅当安全评审点名 Argon2id 时换 **Konscious**（同为纯托管）。
- **权限身份模型**：`CustomIdentity:IIdentity`(含 Roles/SalesPersonKey) + `CustomPrincipal:IPrincipal`(实现 `IsInRole`)，启动挂空 principal，登录成功替换（Magnus Montin 经典范式）。⚠️ `[PrincipalPermission]` 特性在 .NET 9 已废弃，一律改**运行时 `IsInRole` 判断 + 导航守卫**。
- **登录流程**：挂空 principal → 显示登录窗 → `BCrypt.Verify`（失败计数锁定）→ 填 `CurrentUser` 单例 → `MustChangePassword=1` 强制改密 → 按 Role 导航。
- **"记住我"**：用 **DPAPI（`System.Security.Cryptography.ProtectedData`）** 加密令牌（Windows 原生、离线、绑当前用户），**绝不明文存密码**。

### 4.2 数据分区（销售只看/改本人记录）— 硬约束在数据层
在**仓储层集中注入 owner 过滤**，禁止在各 ViewModel 散落 `if role==`：
```csharp
IContractRepository.Query():
   Role==Sales    → .Where(c => c.SalesPersonKey == CurrentUser.SalesPersonKey)   // 不可绕过
   Role>=Manager  → 不加过滤
```
写操作同理校验 `entity.SalesPersonKey == CurrentUser.SalesPersonKey || Role>=Manager`。

### 4.3 个人达成页信息架构（复用 Core StatsEngine，几乎零改造）

**关键：StatsEngine 一行不用改**——它接收一个合同集合算统计，个人页只是传"已按 owner 过滤后的集合"：
- 销售：`engine.Compute(repo.Query())`（repo 已强制 owner 过滤）
- 经理下钻：`engine.Compute(all.Where(c => c.SalesPersonKey == selected))`

页面结构：
- **KPI 卡片行**：收入S完成率 / 利润U完成率 / 成本T / 距目标差额（双口径沿用现有引擎）。
- **个人漏斗**：本人5类合同层级漏斗（LiveCharts2）。
- **1-12月到款折线/柱状**：本人月度到款（现有月度引擎按 owner 过滤）。
- **我的合同明细 DataGrid**：本人全部合同，S/T/U/状态/月度到款，支持排序筛选导出（导出走地基加密+审计）。

### 4.4 经理视角 vs 销售视角（最专业做法）
- **默认视图由登录角色决定**：经理进「团队总览 Shell」，销售进「我的达成 Shell」。用 `ContentControl + DataTemplate` 按 ViewModel 类型自动选视图，零框架依赖（不引 Prism）。
- **经理下钻 = 复用同一个人页 UserControl**，只把注入的 `ownerFilter` 从"本人"改成"被选销售"，**只读下钻，不改写身份**。
- **不做"角色开关按钮"**（越权风险 + 审计难）。切换/下钻/登录/导出全写加密审计日志。
- 视觉参考（仅取卡片/仪表盘结构，不整套换肤，避免与既有浅色企业风冲突）：HandyControl（⭐7.1k）、MaterialDesignInXaml（⭐16.2k）、AduSkin。

## 5. 功能③ U盘离线协同（Hub-and-Spoke，销售→经理汇总→回下发）

### 5.1 机制：自研 op-log + LWW + 金额强制人工裁决
每业务表带 `RowId/UpdatedAt/UpdatedBy/RowVersion/Site`。合并时逐行 upsert：`(UpdatedAt, Site)` 更新者胜；**但金额字段（S/T/U/月度到款）即便 LWW 判定该覆盖，也不自动写，一律进裁决队列**——这是国企台账"绝不静默丢改金额"的红线。

### 5.2 `.cvpk` 包格式（ZIP 容器，加解密全用 BCL 内置算法，FIPS 可验、无第三方 native）
```
contract.cvpk  (ZIP)
├─ manifest.json     明文小JSON：schemaVersion/packageType(delta|masterReturn)/
│                    originSite/period(如2026-07)/baseVersion/rowCount/createdUtc/
│                    producedBy(email)/payloadSha256
├─ manifest.sig      对 manifest.json 的 ECDsa(P-256,CNG) 签名(base64)
└─ payload.bin       AES-256-GCM 密文：nonce(12B)|ciphertext|tag(16B)
                     明文 = changes.jsonl（op-log 每行一条变更）
```
- 加密：`System.Security.Cryptography.AesGcm`（AES-256-GCM 含完整性 tag）；密钥由口令经 `Rfc2898DeriveBytes`(PBKDF2-SHA256 高迭代) 派生。
- 签名：内置 `ECDsa`（P-256）。Ed25519（NSec/minisign-net）仅备选。
- 打包：`System.IO.Compression.ZipArchive`（BCL 内置）。

### 5.3 合并算法步骤（经理侧）
1. **验签**：`ECDsa.VerifyData(manifest, sig, senderPubKey)` 失败→拒绝并审计。
2. **校验**：`SHA256(payload)==manifest.payloadSha256` 失败→拒绝。
3. **解密**：`AesGcm.Decrypt` → `changes.jsonl`。
4. **BEGIN TRANSACTION**，逐行：
   - `cur == null` → INSERT 新行；
   - 否则对每列：金额字段 `AMOUNT_FIELDS` 且值不同 → `enqueue Conflict(rowId,col,主库值,来包值)`（强制人工，默认无选中）；非金额字段 → LWW 自动覆盖；
   - 写 `AuditLog`（来源期次/base/操作/结果）。
5. **COMMIT** → 呈现裁决 UI（DiffPlex 三方对比 + 金额 DataGrid 三列「主库值｜来包值｜采纳值」单选）→ 人工点选后二次事务落库。
6. 记 `Imports(site, period)` → 刷新"谁没交"清单。

### 5.4 裁决 UI：DiffPlex.Wpf（⭐1.3k，Apache-2.0，目标框架含 .NET 9）
- `DiffPlex.Wpf` 是**原生 WPF 控件**（`SideBySideDiffViewer`/`InlineDiffViewer`），符合"不嵌网页"红线。
- 核心库有 `ThreeWayDiffer`（base + 两修改版），正好对应"主库/来包/建议值"三方。
- 金额行用自制 DataGrid 三列 + 单选，**默认无选中，强制点选才可应用**，杜绝静默覆盖；文本/备注类用 DiffPlex 行内高亮。

### 5.5 "谁没交" + 双向下发
- **未交清单**：`期望 Sites×Periods − 已到 Imports` 纯 SQL 反连接（谁/哪期/逾期天数）。
- **上行**：`delta.cvpk` = 自 `baseVersion` 起本站 op-log 变更行。
- **下行**：经理裁决后出 `masterReturn.cvpk`（主版本快照/差量 + 新 `baseVersion`）；销售导入后**以经理主版本覆盖本地**并前移 `baseVersion`，下轮 delta 基于新 base。闭环。

## 7. 建议实现顺序与里程碑

顺序原则：**地基先行（所有块都依赖它），再录入（价值最高、能立刻用），再个人页（复用引擎，改动最小），最后协同（最复杂、依赖前三块的数据/审计/加密）。**

| 里程碑 | 内容 | 交付验收 |
|---|---|---|
| **M0 地基**（约3–4人日） | SQLite3MC 加密库落地 + EF Core 模型/迁移/`AuditableEntity`/软删拦截器/`Version` 触发器 + `CurrentUser` 单例；NPOI `ImportResult`→451条首次灌库 | 加密库能开关、审计/软删/并发生效、灌库幂等、生成首个 baseVersion |
| **M1 类Excel录入**（约3–4人日） | 引 DataGridExtensions；下拉列/计算列/`INotifyDataErrorInfo` 红标；Excel 块粘贴；Ctrl+D 填充；UndoManager | 从 Excel 整块贴入、非法值红标、U=S−T 实时算、撤销重做可用 |
| **M2 登录+个人页**（约2–3人日） | BCrypt 登录 + 强制改密 + DPAPI 记住我；仓储层 owner 过滤；ContentControl 角色导航；个人达成页复用 StatsEngine；经理下钻 | 销售只见本人数据（数据层不可绕过）；经理下钻同组件；KPI/漏斗/月度正确 |
| **M3 U盘协同**（约4–6人日） | op-log 记录；`.cvpk` AES-GCM+ECDsa 打包/验签；LWW 合并 + 金额裁决队列；DiffPlex 裁决 UI；未交清单；双向下发 | 销售导出 delta、经理导入验签解密、金额冲突强制人工、回下发闭环、审计留痕 |
| **M4 收尾（二期可选）** | 填充柄拖拽；导出加密报表；打磨 | — |

## 8. 开工前需用户（朴东旭）拍板的关键点（尽量少，能默认的已默认）

1. **数据库主密钥怎么定？** 默认方案：口令由使用者输入 → PBKDF2 派生库密钥（每人一份加密库）。若国企要求统一密钥托管/密钥找回策略，需你确认。（其余全默认）
2. **协同签名的信任分发**：默认经理侧维护一份"销售公钥指纹信任列表"，销售首次交包时线下核对指纹。是否接受这种线下建信任？（否则要设计密钥分发流程）
3. **安全评审是否点名要 Argon2id**？默认用 BCrypt（工作因子12）。若评审强制 Argon2id，换 Konscious（不影响其他设计）。
4. **是否接受填充柄拖拽放到二期**，一期先用 Ctrl+D 向下填充？（默认：接受）
5. **是否确认不采购商用控件**（全程原生 DataGrid）？（默认：不采购；仅当二期交互自写成本过高才评估 Syncfusion SfDataGrid 走采购）

以上 5 点之外的所有选型与设计均已给默认值，可直接开工 M0。
